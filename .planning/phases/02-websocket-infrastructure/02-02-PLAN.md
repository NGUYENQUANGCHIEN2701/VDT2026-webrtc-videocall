<objective>
Implement realtime online-presence tracking and broadcast. Every STOMP connect and disconnect must (a) update an in-memory session registry, (b) flip the user's DB `status` column to ONLINE / OFFLINE (PRES-02), and (c) publish the current online user list to `/topic/presence` so all subscribers receive it without polling (PRES-01).

Purpose: PRES-01 and PRES-02 are the two requirements that turn a "users connect to WebSocket" capability into a "users can see who is online" feature. Plan 02-01 made the broker work; this plan makes the broker observe its own connection lifecycle and broadcast it. Plan 02-03 then layers signaling on top of an already-presence-aware system. Without this plan, Phase 3 (React user list) has nothing to subscribe to.

Output: One new @Service, one new @Component event listener, one new DTO, and the previously-disabled `PresenceBroadcastTest` driven from RED (4 skipped) to GREEN (2 passing).
</objective>

    Add `@EventListener` method `onConnect(SessionConnectedEvent event)`:
      1. Wrap the event message: `StompHeaderAccessor accessor = StompHeaderAccessor.wrap(event.getMessage())`
      2. Get `Principal user = accessor.getUser()`. If null, log a warning ("SessionConnectedEvent without principal — interceptor ordering issue, see Pitfall 4") and `return` — do NOT throw.
      3. Capture `String username = user.getName()` and `String sessionId = accessor.getSessionId()`
      4. Call `presenceService.addSession(sessionId, username)`
      5. Flip DB status: `userRepository.findByUsername(username).ifPresent(u -&gt; { u.setStatus(UserStatus.ONLINE); userRepository.save(u); })`
      6. Call `broadcastPresence()`

    Add `@EventListener` method `onDisconnect(SessionDisconnectEvent event)`:
      1. Wrap the message AND capture sessionId first (sessionId is always present): `String sessionId = event.getSessionId()`
      2. `StompHeaderAccessor accessor = StompHeaderAccessor.wrap(event.getMessage())` and `Principal user = accessor.getUser()` — but treat user as @Nullable per Pitfall 4 and event.getUser()'s @Nullable contract

      4. If `user != null`:
         - `String username = user.getName()`
         - After session removal, check `if (!presenceService.isUserOnline(username))`. If true (user has no remaining sessions), flip DB status: `userRepository.findByUsername(username).ifPresent(u -&gt; { u.setStatus(UserStatus.OFFLINE); userRepository.save(u); })`. This guards against Pitfall 3 (duplicate disconnect events) and the "multiple tabs" scenario.
      5. Call `broadcastPresence()` (always — even if user was null, a session might have been tracked)

    Private helper `void broadcastPresence()`:
      - `List&lt;String&gt; onlineUsers = presenceService.getOnlineUsers()`
      - `messagingTemplate.convertAndSend("/topic/presence", new PresenceDTO(onlineUsers))`

    Do NOT inject `SimpMessagingTemplate` by qualifier — by-type resolution works because `@EnableWebSocketMessageBroker` exposes exactly one `SimpMessagingTemplate` bean. Do NOT add `@Async` to the listeners — Spring's STOMP event publication is already on the broker channel; @Async would reorder events. Do NOT call `userRepository.save` inside the message thread without the existing JPA transaction — Spring Data JPA auto-wraps `.save()` in a transaction so no `@Transactional` annotation is needed at the listener level (matches Phase 1 AuthService pattern).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| STOMP CONNECT → SessionConnectedEvent | Authenticated principal flows from the JWT interceptor (Plan 02-01) into the event listener |
| Event listener → /topic/presence | Authenticated server-side broadcast; every subscriber receives the payload regardless of subscriber identity |
| Event listener → UserRepository.save | Privileged DB write; only username pulled from the JWT principal can drive a status change |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-PI | Information Disclosure | /topic/presence broadcast contains usernames of all online users | accept | Authenticated subscribers only (CONNECT was JWT-validated in Plan 02-01); usernames are not secrets in this product (REQUIREMENTS.md PRES-01 explicitly exposes them) |
| T-2-PS | Spoofing | Client-supplied username via SessionConnectedEvent payload | mitigate | The listener reads username from `accessor.getUser().getName()` — the principal set by JwtChannelInterceptor — never from a client-supplied frame header |
| T-2-PR | Repudiation / data integrity | Duplicate SessionDisconnectEvent flips DB status repeatedly | mitigate | Idempotency guard: DB OFFLINE write is gated on `!presenceService.isUserOnline(username)` so a second event finds the session already removed and skips the save (Pitfall 3) |
| T-2-PD | Denial of Service | Flood of connect/disconnect events causing broadcast storms | accept | LAN demo scope; broker is in-memory and `ConcurrentHashMap` ops are O(1); 02-RESEARCH.md security domain explicitly accepts this for MVP |
| T-2-PN | Tampering | Null principal on SessionConnectedEvent due to interceptor ordering misconfig | mitigate | Listener null-checks `accessor.getUser()` and logs a warning instead of NPE; root cause (interceptor ordering) is mitigated in Plan 02-01 via @Order(HIGHEST_PRECEDENCE + 99) |
| T-2-PT | Tampering | Listener writes UserStatus from a stale entity | accept | `findByUsername` reads the row fresh inside the same Spring-managed transaction; no optimistic-lock collision expected on a single column in MVP |
</threat_model>
