<objective>
Add the signaling relay layer that lets one authenticated client send a SDP/ICE/call-control message to a named recipient via STOMP. Implements CALL-05 ("signaling via WebSocket (STOMP) exchanging SDP offer/answer and ICE candidates") and provides the integration tests that prove the spoof-prevention guarantee from the Phase 2 success criteria ("message to /app/signal → routed to recipient's /user/queue/signal using JWT principal").

CALL-06 (Google public STUN) is a frontend-only configuration of `RTCPeerConnection.iceServers`; the backend is intentionally opaque about payload semantics. This plan still claims requirement CALL-06 ownership because the relay design (treating `payload` as an opaque string) is what makes the eventual ICE candidate exchange in Phase 4 work without backend changes. The "implementation" of CALL-06 in this phase is a negative claim: the controller never parses, validates, or modifies the `payload` field.

Output: One new @Controller, one new DTO, and the previously-disabled `SignalRelayTest` driven from RED (2 skipped) to GREEN (2 passing). After this plan, Phase 2 ships GREEN and Phase 3 can build the React user list against a fully presence-aware, signaling-capable backend.
</objective>

    Add method:
    ```
    @MessageMapping("/signal")
    public void handleSignal(@Payload SignalMessage message, java.security.Principal principal)
    ```
    Body in directive form:
      1. If `principal == null` → `log.warn("Signal received without authenticated principal — dropping")` and `return`. (Spoofing defense — should not happen because interceptor rejects unauthenticated CONNECT, but defense-in-depth.)
      2. If `message == null || message.getTo() == null || message.getTo().isBlank()` → `log.warn("Signal from {} missing recipient — dropping", principal.getName())` and `return`.
      3. Overwrite the spoofable field: `message.setFrom(principal.getName())`. This MUST happen before the relay call. (T-2-02 spoofing mitigation — even if attacker sets from="admin" in JSON, server replaces it.)
      4. `log.debug("Signal from {} to {} type={}", message.getFrom(), message.getTo(), message.getType())` — log type for debugging the offer/answer/ICE sequence; never log `message.getPayload()` (raw SDP can leak network info).
      5. Relay: `messagingTemplate.convertAndSendToUser(message.getTo(), "/queue/signal", message)` — destination is RELATIVE (no `/user/` prefix) per Pitfall 6. Spring's UserDestinationMessageHandler will route to `/user/{to}/queue/signal`.

    Do NOT add `@SendToUser` annotation — `convertAndSendToUser` is the explicit form and we want it for clarity and for the ability to drop the relay (early returns above). Do NOT introduce an interface for the message handler — direct controller is the Spring-canonical pattern (Research Pattern 6).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client SEND to /app/signal | Authenticated session can supply arbitrary JSON body; controller must not trust the `from` field |
| server-side relay → /user/{to}/queue/signal | Spring's UserDestinationMessageHandler translates the relative destination using the authenticated session's principal table; only sessions belonging to user `{to}` receive the frame |
| opaque `payload` field | Carries SDP and ICE candidates in Phase 4; backend must never parse this — keeps CALL-06 (Google STUN) entirely frontend-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-02 | Spoofing | SignalMessage.from field client-supplied | mitigate | SignalController unconditionally executes `message.setFrom(principal.getName())` BEFORE relay; grep-gate acceptance criterion 5 of Task 1 enforces the literal call presence; testSignalFromOverwritten asserts behavior |
| T-2-SD | Tampering | Wrong destination prefix → undeliverable messages | mitigate | Controller passes RELATIVE destination `"/queue/signal"` to convertAndSendToUser (Pitfall 6); grep-gate acceptance criterion 7 of Task 1 forbids the `/user/queue/signal` form |
| T-2-PD | Information Disclosure | SDP payload contains private LAN IP addresses (host candidates) | accept | LAN demo scope; ICE candidate exchange between trusted peers is the entire point of WebRTC signaling; payload is end-to-end between two authenticated peers, no third-party leakage |
| T-2-SP | Information Disclosure | Raw payload logged | mitigate | Controller logs only `type` field, never `payload`; grep-gate acceptance criterion 8 of Task 1 enforces |
| T-2-DR | Denial of Service | Flood of /app/signal frames | accept | LAN demo scope; per 02-RESEARCH.md Security Domain — "no per-connection rate limiting in MVP" is acceptable |
| T-2-RT | Spoofing | Send /app/signal without principal (regression in Plan 02-01) | mitigate | Defense-in-depth — controller null-checks principal and drops with warn; primary mitigation is the JwtChannelInterceptor from Plan 02-01 |
| T-2-VR | Tampering | Sender targets a non-existent or offline `to` user | accept | convertAndSendToUser to an unknown user is silently dropped by the simple broker; per Open Question 2 of 02-RESEARCH.md, error feedback is deferred to Phase 4 call state management |
</threat_model>
