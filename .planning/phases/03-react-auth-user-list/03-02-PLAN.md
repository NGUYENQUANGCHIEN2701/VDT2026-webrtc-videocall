<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form → Axios → /api/auth/login,/register,/logout | Untrusted credentials cross the LAN to the backend |
| Browser → STOMP /ws (post-login) | JWT presented in STOMP CONNECT headers |
| AuthPage component → localStorage | LOGIN dispatch writes the token |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-07 | Information Disclosure | Password field in DOM | LOW | mitigate | Input type='password' (UI-SPEC §5). No logging of credentials in AuthPage submit handler (no `console.log(password)`). Form state cleared from React state on successful navigation. |
| T-3-08 | Tampering | Form input injection (XSS via username) | MEDIUM | mitigate | All form values flow through controlled React `<Input>` components (escaped automatically by React's text rendering). No `dangerouslySetInnerHTML`. Error messages from API are rendered as text inside `<AlertDescription>` (also escaped by React). |
| T-3-09 | Information Disclosure | JWT echoed to network logs | LOW | mitigate | Axios interceptor (Plan 01) injects the token only in the `Authorization` header. AuthPage never places the token in URL, query string, or DOM. The token is only in: response body, localStorage, AuthContext state, STOMP connectHeaders. |
| T-3-10 | Spoofing | Stale token from prior session | LOW | mitigate | AuthPage relies on AuthContext LOGIN action to overwrite localStorage; the dispatched LOGIN always sets the new token. WebSocketContext.connect always uses the fresh token argument, not closure over old state. |
| T-3-11 | Information Disclosure | Backend error leaks (e.g., stack trace surfaced to UI) | LOW | mitigate | AuthPage maps HTTP status → fixed user-facing copy per UI-SPEC §5; backend response body is not rendered. Phase 1 GlobalExceptionHandler already sanitizes error payloads. |
| T-3-12 | Denial of Service | Repeated submit while loading | LOW | mitigate | Submit button is `disabled` while `isLoading=true`; double-click cannot fire a second request. |

**HIGH severity items:** none. T-3-08 (XSS) is MEDIUM but fully mitigated by React's default text-escaping and the absence of any innerHTML/eval/dangerouslySetInnerHTML in the component (verified by acceptance grep).

</threat_model>
