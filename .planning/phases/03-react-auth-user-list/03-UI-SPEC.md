| Property | Value |
|----------|-------|
| Component library | shadcn/ui (official registry only) |

### shadcn/ui Initialization

Run once at project setup. This is the exact command developers must use:

```bash
npx shadcn@latest init
```

Answers to `shadcn init` prompts:

| Prompt | Answer |
|--------|--------|
| Which style would you like to use? | Default |
| Which color would you like to use as the base color? | Slate |
| Would you like to use CSS variables for theming? | Yes |

After init, apply the dark theme override in `src/index.css` — see Color section below.

### Tailwind Config Additions

After shadcn init, extend `tailwind.config.js` to ensure emerald is available as a named color:

```js
// tailwind.config.js — extend only, do not replace defaults
theme: {
  extend: {
    colors: {
      // shadcn CSS vars are already wired; these are convenience aliases
      accent: {
        DEFAULT: '#10B981',   // emerald-500
        hover:   '#059669',   // emerald-600
        muted:   '#064E3B',   // emerald-900 (for ghost backgrounds)
      }
    }
  }
}
```

## 2. Spacing Scale

All spacing tokens are multiples of 4px. Developers must not use arbitrary pixel values outside this table.

| Token | Tailwind class | Value | Usage |
|-------|---------------|-------|-------|
| xs | `p-1` / `gap-1` | 4px | Icon internal padding, badge text gap |
| sm | `p-2` / `gap-2` | 8px | Compact row gaps, icon-to-label gaps |
| md | `p-4` / `gap-4` | 16px | Default element padding, form field gaps |
| lg | `p-6` / `gap-6` | 24px | Card internal padding, section spacing |
| xl | `p-8` / `gap-8` | 32px | Card-to-edge margin, layout section gaps |
| 2xl | `p-12` / `gap-12` | 48px | Auth card vertical centering offset |
| 3xl | `p-16` / `gap-16` | 64px | Page-level vertical centering |

Exceptions: none.

## 5. Auth Screen Contract

### Layout

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│              [App logo / name]                      │
│           VDT-WebRTC (text mark)                   │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │ ┌─────────────────┐ ┌─────────────────────┐  │  │
│  │ │   Login         │ │   Register          │  │  │
│  │ └─────────────────┘ └─────────────────────┘  │  │
│  │ ─────────────────────────────────────────── │  │
│  │  [Form fields for active tab]               │  │
│  │  [Submit button]                            │  │
│  └───────────────────────────────────────────────┘  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

- Page: `min-h-screen` with `flex items-center justify-center` on `bg-slate-950`
- Card: `w-full max-w-md` with `bg-slate-900 border border-slate-700 rounded-xl shadow-2xl`
- Card inner padding: `p-8`
- Logo area: centered, above card — `Video` icon (lucide) at `size-8` + `VDT-WebRTC` in `text-xl font-semibold text-slate-50`

### Tab Toggle

Component: `shadcn/ui Tabs` (`TabsList` + `TabsTrigger` + `TabsContent`)

| State | Visual |
|-------|--------|
| Active tab | `text-emerald-400`, bottom border `border-b-2 border-emerald-500`, `bg-transparent` |
| Inactive tab | `text-slate-400`, no border, hover: `text-slate-200` |
| TabsList | `bg-slate-800 rounded-lg p-1 w-full grid grid-cols-2` |

Tab labels: **Login** / **Register** (exact casing — title case).

Tab switch transition: `transition-all duration-200 ease-in-out` on `TabsContent` via `data-[state=active]:animate-in data-[state=inactive]:animate-out` (shadcn default). No custom animation required — shadcn handles it.

### Login Tab — Form Fields

| Field | Component | Type | Placeholder | Validation |
|-------|-----------|------|-------------|-----------|
| Username | shadcn `Input` | `text` | `Enter your username` | Required, min 3 chars |
| Password | shadcn `Input` | `password` | `Enter your password` | Required, min 6 chars |

Layout: `space-y-4` between field groups. Each field group: `Label` above `Input`.

Label text: **Username** / **Password** (exact casing). Labels use `text-sm font-normal text-slate-300`.

Submit button: full-width `Button` variant `default` — `w-full bg-emerald-500 hover:bg-emerald-600 text-white font-semibold h-10`. Label: **Sign In**.

### Register Tab — Form Fields

| Field | Component | Type | Placeholder | Validation |
|-------|-----------|------|-------------|-----------|
| Username | shadcn `Input` | `text` | `Choose a username` | Required, min 3 chars, max 32 chars |
| Password | shadcn `Input` | `password` | `Create a password` | Required, min 6 chars |
| Confirm Password | shadcn `Input` | `password` | `Confirm your password` | Must match password field |

Submit button: **Create Account**. Same styling as Sign In button.

### Validation States

| State | Visual |
|-------|--------|
| Field untouched | `border-slate-700` (default) |
| Field focused | `border-emerald-500 ring-1 ring-emerald-500/50` (via `--ring` CSS var) |
| Field error | `border-red-500 ring-1 ring-red-500/50` + error message below |
| Field valid (on blur, after first submit attempt) | `border-slate-700` (reset — do not show green tick in Phase 3) |

### Error Messages (Inline)

Displayed immediately below the offending field. Component: `<p className="text-xs text-red-400 mt-1">`.

| Trigger | Message |
|---------|---------|
| Username empty | `Username is required` |
| Username too short | `Username must be at least 3 characters` |
| Password empty | `Password is required` |
| Password too short | `Password must be at least 6 characters` |
| Confirm password mismatch | `Passwords do not match` |
| Login failed (401 from API) | `Invalid username or password` |
| Register failed — username taken (409 from API) | `Username already taken. Please choose another.` |
| Network / server error (5xx) | `Something went wrong. Please try again.` |

API errors: displayed in a `shadcn/ui Alert` variant `destructive` at the top of the form (above first field), not inline. Example:

```
⚠  Invalid username or password
```

### Loading State

While API call is in flight:

- Submit button: `disabled` + replace label with `<Loader2 className="animate-spin mr-2 h-4 w-4" /> Signing in...` (Login) or `Creating account...` (Register)
- All form inputs: `disabled` (pointer-events-none via prop)
- No skeleton overlay — just button state is sufficient

## 6. Online User List Screen Contract

### Layout

```
┌────────────────────────────────────────────────────────┐
│  Header                                                │
│  ┌──────────────────────────────────────────────────┐  │
│  │  [Video icon]  VDT-WebRTC    [username]  Logout  │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  Content area (scrollable)                             │
│  ┌──────────────────────────────────────────────────┐  │
│  │  Online Users   (N online)                       │  │
│  │  ────────────────────────────────────────────    │  │
│  │  [Avatar] Alice       ● Online    [Call]         │  │
│  │  [Avatar] Bob         ● Online    [Call]         │  │
│  │  [Avatar] Carol       ● Online    [Call]         │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

### Header

- Fixed at top: `sticky top-0 z-10 bg-slate-900/80 backdrop-blur-sm border-b border-slate-700`
- Height: `h-14` (56px)
- Inner: `flex items-center justify-between px-6`
- Left: `Video` icon (lucide, `size-5 text-emerald-400`) + `VDT-WebRTC` in `text-base font-semibold text-slate-50`
- Right: username in `text-sm text-slate-400` + `|` divider + **Logout** button

Logout button: shadcn `Button` variant `ghost` size `sm`. Label: **Logout**. Icon: `LogOut` (lucide, `size-4 mr-1`). On click: shows confirmation inline or proceeds directly — Phase 3 proceeds directly (no confirm dialog). POST to `/api/auth/logout`, then clear JWT from state and navigate to auth screen.

### User List Panel

- Container: `mx-auto max-w-2xl px-4 py-6` (centered, readable max-width on large screens)
- Panel card: `bg-slate-900 border border-slate-700 rounded-xl overflow-hidden`
- Panel header: `px-6 py-4 border-b border-slate-700 flex items-center justify-between`
  - Left: `Online Users` in `text-base font-semibold text-slate-50`
  - Right: count badge — `N online` in `text-xs font-normal text-emerald-400 bg-emerald-900/30 px-2 py-1 rounded-full`
- List: `divide-y divide-slate-700/50` — no external list component, plain `ul`/`li`

### User Row

Each online user renders one `li` row:

```
┌──────────────────────────────────────────────────────────┐
│  [Avatar]   Alice             ● Online        [Call]     │
│  40px       text-sm           badge           button     │
└──────────────────────────────────────────────────────────┘
```

- Row: `flex items-center gap-4 px-6 py-3 hover:bg-slate-800/50 transition-colors duration-150`
- Avatar: shadcn `Avatar` — `AvatarFallback` with first letter of username, `bg-slate-700 text-slate-200 text-sm font-semibold`, size `h-10 w-10`
- Username: `text-sm font-normal text-slate-50` — truncated with `truncate max-w-[160px]`
- Spacer: `flex-1`
- Status badge: `● Online` — `text-xs font-normal text-emerald-400`, with `●` rendered as `•` (U+2022) in emerald
- Call button: shadcn `Button` variant `default` size `sm` — `bg-emerald-500 hover:bg-emerald-600 text-white`
  - Icon: `Phone` (lucide, `size-4 mr-2`)
  - Label: **Call**
  - `aria-label`: `Call {username}`

The current logged-in user must NOT appear in their own user list. Filter on the client side using the stored JWT subject.

### Empty State

Shown when no other users are online (list is empty after filtering self).

- Centered within the panel content area: `flex flex-col items-center justify-center py-16 text-center`
- Icon: `Users` (lucide, `size-12 text-slate-600 mb-4`)
- Heading: `No one else is online` in `text-base font-semibold text-slate-400`
- Body: `Share the app link with a friend to start a call.` in `text-sm text-slate-500`
- No CTA button in empty state — Phase 3 does not include invite flow

### Loading State

Shown on initial page load before the first WebSocket presence message arrives.

- Render 3 skeleton rows using shadcn `Skeleton`:

```jsx
// Skeleton row (repeat 3x)
<div className="flex items-center gap-4 px-6 py-3">
  <Skeleton className="h-10 w-10 rounded-full" />
  <Skeleton className="h-4 w-32" />
  <div className="flex-1" />
  <Skeleton className="h-6 w-16 rounded-full" />
  <Skeleton className="h-8 w-16 rounded-md" />
</div>
```

- Loading resolves when the first `/topic/presence` STOMP message is received (or after 3s timeout — show empty state as fallback)

### WebSocket List Updates

When the presence subscription receives an updated user list:

- New users appearing: row slides in from the top with `animate-in fade-in slide-in-from-top-1 duration-300`
- Users going offline: row fades out with `animate-out fade-out duration-200` before removal
- Implementation: use React `key={user.id}` on each list item to let React reconcile naturally; CSS transitions handle the visual effect via Tailwind animation utilities

## 10. Accessibility Requirements

| Requirement | Implementation |
|-------------|----------------|
| All interactive elements keyboard-navigable | shadcn components use Radix primitives — keyboard nav is built-in |
| Tab order is logical | Auth card: tab → username → password → submit. Register: tab → username → password → confirm → submit |
| Error messages announced to screen readers | Use `aria-describedby` linking each input to its error `<p>` element |
| Submit button disabled state announced | `aria-disabled="true"` and `disabled` prop on Button during loading |
| Call button has accessible label | `aria-label="Call {username}"` — button text alone ("Call") is insufficient with screen readers |
| Focus ring visible | `--ring` CSS variable set to emerald-500; all shadcn inputs show it on focus |
| Color contrast — primary text on card | `text-slate-50` (#F8FAFC) on `bg-slate-900` (#0F172A) = 16.6:1 — AAA |
| Color contrast — muted text | `text-slate-400` (#94A3B8) on `bg-slate-900` (#0F172A) = 5.9:1 — AA |
| Color contrast — emerald text on dark | `text-emerald-400` (#34D399) on `bg-slate-900` (#0F172A) = 6.8:1 — AA |
| Headings use semantic HTML | `<h1>` for app name, `<h2>` for panel title, not just styled `<div>` |
| Form has semantic structure | `<form>` element with associated `<label>` elements using `htmlFor` |
| Auth card dialog-like region | Wrap card content in `<main>` with `aria-label="Authentication"` |
| User list region | Wrap list in `<section aria-label="Online users">` |
| Avatar image alternative | `AvatarFallback` provides text; no `<img>` in Phase 3 so no `alt` needed |
| Skeleton loading announcement | `<div role="status" aria-label="Loading online users...">` wrapping skeleton rows |

WCAG target: **2.1 Level AA**. All color contrast ratios above meet AA minimum (4.5:1 for normal text).

## 11. Out of Scope for Phase 3

The following items are explicitly NOT implemented in Phase 3, even if they appear tempting during development:

| Feature | Why deferred |
|---------|-------------|
| Light mode toggle | Dark-only for Phase 3; multi-theme adds config complexity before core features work |
| User avatar image upload | No storage backend; `AvatarFallback` with initials is sufficient |
| Incoming call modal / ringtone | Phase 4 — requires WebRTC call flow which doesn't exist yet |
| Call screen / video streams | Phase 4 |
| Mic/camera permission prompt | Phase 4 |
| Password strength meter | Not in requirements; adds scope without value |
| "Remember me" / persistent JWT | sessionStorage only in Phase 3 — no refresh token until v2 |
| Offline user list (users who logged out) | Presence only shows online users; offline state is simply absent |
| Search / filter in user list | Not in requirements for Phase 3 |
| User profile page | Out of scope (REQUIREMENTS.md) |
| Toast notifications for list updates | Phase 4+ — list animation is sufficient visual feedback in Phase 3 |
| Mobile responsive layout | Out of scope (REQUIREMENTS.md — desktop browser only) |
| Dark/light toggle or system preference | Dark only; reduces complexity for demo deliverable |
| Route guards / redirect logic beyond auth | Phase 3 has 2 routes: `/auth` and `/users` — simple `PrivateRoute` wrapper only |
